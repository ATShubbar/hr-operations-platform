## Implementation notes (Prisma 7 specifics — read before WS-12/13)

- **Prisma resolved to v7.8**, which changed the model vs. v6: the datasource URL is **not** allowed in `schema.prisma`; it lives in `prisma.config.ts` (`defineConfig({ datasource: { url: env('DATABASE_URL') } })`). The runtime client requires a **driver adapter** — we use `@prisma/adapter-pg` constructed straight from the connection string.
- **Telemetry hang:** `prisma migrate dev`/`generate` hang after completing their work (checkpoint/telemetry HTTP call). All db scripts set `CHECKPOINT_DISABLE=1`. A 3-minute hang mid-task was diagnosed as exactly this — the migration itself had applied fine.
- **Generated client is committed** (`src/generated/prisma/`, new v7 generator emits TS source). Rationale: CI runs typecheck/build without a DB or generate step. Excluded from ESLint (`ignores`). Revisit if regeneration churn becomes noisy.
- **Throwaway table `ws11_check` + its two migrations get removed** when the first real module table lands (WS-13 or Priority-2 Clients) — squashing init migrations pre-production is acceptable since no shared DB exists yet.
- Driver-adapter + `PrismaPg` accepting a `pg.PoolConfig` is directly relevant to WS-12: pool control is in our hands, which changes the spike's pooling questions in our favor.
